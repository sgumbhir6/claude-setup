# One-Pager: Proposal

## Stakeholders
Engineers and Product Managers

## Tone
Accessible to both technical and non-technical readers. Use service names and business impact over code-level details. Be concise and direct.

## Sections

### Context
Why we need this solution. What problem exists today, what this solution helps us gain, why it matters now.

**Source:** Read `problem-statement.md` for the human-readable framing. If technical constraints are relevant to the "why", also read `agent-context.md`.

### Proposed Solution
The main body of the document. Should contain sub-headings with details about the solution.

**[ask user for sub-sections]** — Before writing this section, ask the user: "What sub-sections should the Proposed Solution include?" Examples: architecture changes, rollout plan, migration strategy, API changes, etc.

**Source:** Read `findings.md` for evidence that supports the proposal. Read `agent-context.md` for technical constraints that shape the solution.
