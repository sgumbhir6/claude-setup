# One-Pager: Investigation

## Stakeholders
Engineers

## Tone
Technical, evidence-based. Can use service names, code-level concepts, and engineering terminology freely. Be precise and concise.

## Sections

### Problem Statement
What was investigated. What is the observed behavior, what is the expected behavior, what is the impact.

**Source:** Read `problem-statement.md` for the framing. Read `agent-context.md` for technical scope and affected services.

### Findings
What was discovered during the investigation. Key evidence organized logically — not chronologically. Each finding should state what was observed and what it means.

**Source:** Read `findings.md` for the evidence. Reference specific behaviors and services, but do not include raw file paths or line numbers — this is a business doc, not a code review.

### Proposed Resolution
What should be done about it. Concrete recommendation based on the findings.

**Source:** Read `findings.md` for evidence that informs the recommendation. Read `agent-context.md` for constraints that shape what is feasible.
